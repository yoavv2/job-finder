<objective>
Build the two reusable resume output/validation stages on top of Plan 04's structured model:

1. **HTML -> PDF renderer (RES-02):** `renderResumeHtml(resume)` turns a structured Resume into single-column, ATS-readable, semantic HTML; `renderResumePdf(html)` renders that HTML to a selectable-text PDF via headless Chromium (Puppeteer). The pipeline is explicitly `structured -> HTML -> PDF`, as two reusable stages. PDFs are output-only.

2. **Deterministic integrity validator (RES-03):** `validateResumeIntegrity(master, tailored)` performs an entity-diff over the typed Resume model and rejects any company, technology/skill, project, or claim that appears in the tailored resume but is absent from the master. This is the **mechanical, non-prompt** integrity guarantee Phase 4 enforces before rendering — deterministic, no LLM, no PDF parsing.

This is substrate only: do NOT build the Phase 4 per-job LLM tailoring agent here. Build the reusable renderer + validator it will call.

Purpose: Trustworthy resume output + the mechanism that makes "never fabricate" structural rather than hoped-for.
Output: html.ts, pdf.ts, integrity.ts + tests; Puppeteer added to deps.
</objective>

  <name>Task 1: Deterministic structured-vs-structured integrity validator (entity-diff)</name>
  <files>src/resume/integrity.ts, src/resume/integrity.test.ts</files>
  <behavior>
    RED first. integrity.test.ts (uses small hand-built master + tailored Resume objects):
    - Identical master/tailored (or tailored is a strict reorder/rewrite-of-summary of master's entities) -> { ok: true, violations: [] }.
    - Tailored introduces a company in experience not present in master -> ok:false, a violation naming that company (category 'company').
    - Tailored introduces a skill/technology (in skills or in a project's technologies) absent from master -> ok:false, violation category 'technology'.
    - Tailored introduces a project name absent from master -> ok:false, violation category 'project'.
    - Reordering skills and rewriting the summary wording (no new proper-noun entities) -> ok:true (tailoring is allowed to reorder/rephrase; only NEW entities are rejected).
    - Determinism: calling twice on the same inputs yields identical violations (no LLM, no randomness).
    - Document/assert no PDF parsing and no LLM import in this module.
  </behavior>
  <action>
    Implement `validateResumeIntegrity(master: Resume, tailored: Resume): { ok: boolean; violations: { category: 'company'|'technology'|'project'|'claim'; value: string }[] }` in src/resume/integrity.ts.
    Strategy (deterministic, pure):
    - Build entity sets from the MASTER: companies (experience[].company), technologies (skills items + projects[].technologies), projects (projects[].name). Normalize for comparison (trim + lowercase) but report the original tailored value.
    - Scan the TAILORED resume's corresponding fields; any company/technology/project whose normalized form is NOT in the master's set is a violation of that category.
    - For 'claim': scan tailored experience/project bullets for proper-noun-like tokens (capitalized multi-word phrases or tokens matching known master entities) — at minimum, flag bullets that reference a company/technology/project name not in the master. Keep this conservative and documented; the goal is to reject fabricated NEW entities, not to rewrite-detect (rephrasing existing claims is allowed). Document the chosen claim heuristic in a top-of-file comment.
    - ok = violations.length === 0.
    Pure function: no LLM, no fs, no PDF. Phase 4 will call this before rendering.
  </action>
  <verify>
    <automated>nvm use 22.22.0 && corepack pnpm vitest run src/resume/integrity.test.ts</automated>
  </verify>
  <done>validateResumeIntegrity deterministically passes legitimate reorder/rephrase tailoring and rejects any introduced company/technology/project/claim absent from the master, reporting each violation; no LLM/PDF involved.</done>
